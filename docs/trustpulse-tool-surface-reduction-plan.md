# TrustPulse Tool Surface Reduction Plan

## Objective

Reduce the default tool surface in this `chrome-devtools-mcp` fork so agents stay focused on onboarding/navigation work.

Confirmed requirements:
- Keep screenshot tooling.
- Keep console/network debugging tooling (`get/list console`, `get/list network`) for now.
- Remove the currently low-value tools listed below.

## Scope

### Tools to remove

- `performance_analyze_insight`
- `performance_start_trace`
- `performance_stop_trace`
- `take_memory_snapshot`
- `lighthouse_audit`
- `drag`
- `emulate`

### Tools to keep explicitly

- `list_pages`

- `new_page`
- `navigate_page`

- `click`
- `fill`

- `take_snapshot`
- `take_screenshot`
- `evaluate_script`
- `handle_dialog` (kept for unexpected popups)

Debugging (kept intentionally):
- `list_console_messages`
- `get_console_message`
- `list_network_requests`
- `get_network_request`

## Non-goals

- No changes to the TrustPulse runner in this step.
- No highlight/record API redesign in this step.
- No changes to `--slim` mode behavior.

## Implementation Strategy

### Design choice

Use a central fork-level denylist in tool assembly/registration rather than deleting tool implementation files.

Reason:
- Lowest risk and easiest rollback.
- Keeps upstream merge friction low.
- Preserves code for later re-enable if needed.

### Target files

Primary:
- `src/tools/tools.ts`

Potentially impacted/generated:
- `src/bin/cliDefinitions.ts` (generated)
- `docs/tool-reference.md` (generated)
- `README.md` auto-generated tool section (generated by `npm run gen`)
- `src/telemetry/tool_call_metrics.json` (if regenerated)

Tests likely needing updates:
- tests that snapshot the exposed tool list / docs / CLI outputs.

## Detailed Execution Plan

1. Baseline current tool exposure.
- Run MCP list-tools locally and save before/after comparison.
- Record exact currently exposed tool names.

2. Add denylist filtering in `src/tools/tools.ts`.
- Keep existing module imports to reduce code churn.
- After collecting `rawTools` and materializing `tools`, filter by name against:
  - `performance_analyze_insight`
  - `performance_start_trace`
  - `performance_stop_trace`
  - `take_memory_snapshot`
  - `lighthouse_audit`
  - `drag`
  - `emulate`
  - `resize_page`
- Leave console/network tools untouched.

3. Regenerate generated artifacts.
- Run `npm run gen` so CLI definitions/docs stay consistent with exposed tools.
- Confirm generated docs no longer advertise removed tools.

4. Update or regenerate snapshots/tests.
- Run tests that assert tool catalogs or generated outputs.
- Update snapshots only for expected tool-list deltas.

5. Validate functional behavior manually.
- Start local server and list tools.
- Confirm removed tools are absent.
- Confirm kept tools are present.
- Run a quick onboarding-style smoke sequence with:
  - `navigate_page`
  - `take_snapshot`
  - `click`
  - `take_screenshot`
  - `list_console_messages`
  - `list_network_requests`

6. Prepare rollback path.
- Rollback is single-commit revert restoring previous tool exposure.
- Because implementation files remain, rollback does not require reconstructing deleted code.

## Risks and Mitigations

Risk: Existing automation prompts might still mention removed tools.
- Mitigation: keep prompts focused on core tools; add one-time prompt hygiene check.

Risk: Agents may lose rare debugging capability from removed performance/lighthouse tools.
- Mitigation: console/network debugging tools remain available.

Risk: Upstream merge conflicts in generated files.
- Mitigation: keep custom logic small and centralized in `src/tools/tools.ts`; regenerate deterministically.

## Rollout Notes

- This change is fork-specific and intentionally opinionated for TrustPulse-style onboarding runs.
- If needed later, we can evolve this into a configurable profile flag (for example `--toolProfile trustpulse`) instead of fixed fork defaults.
